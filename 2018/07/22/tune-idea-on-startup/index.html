Idea 启动调优

2018-07-22

说实话，idea 真的是第一次用就喜欢上了，但平时开发都在公司，电脑和代码都不允许带回家，在家写代码的时候很少，所以安装 Community 版足够了，不然早就买了 Ultimate 版。

1、查看启动参数

先安装最新版本，当前是

安装完成后打开启动参数配置文件，我用的是 Mac，可以在 Application 里找到 IntelliJ Idea CE 后鼠标右键选择显示包内容，找到 Contents/bin/idea.vmoptions 文件就是 idea 的启动参数配置文件。打开看一下默认配置，

-Xverify:none 禁止了字节码验证，毕竟 idea 代码的可靠性是经过无数次的运行验证了的，所以启动时这一步可以关闭。最后三行是我加的，其余没有做修改。有几个不太常用的参数，参考了 JVMPocket 等资料，在 JDK8 里的含义如下：

参数 含义
-XX:ReservedCodeCacheSize 和 -Xmaxjitcodesize 等价，用来设置 CodeCache 的最大值，CodeCache 是用来存放 JIT 编译器编译后的代码，JDK8 64 位系统下默认 240M，如果 CodeCache 满了，会输出 CodeCache is full 日志，并禁止即时编译，如果启用 -XX:+UseCodeCacheFlushing 参数，会丢弃一般老的代码，并在可用空间超过 -XX:CodeCacheMinimumFreeSpace=<N> 指定的空间（默认 500K）后启用即时编译。参考。
-XX:+UseCompressedOops 64 位 OS 下有效，启动指针压缩。
-XX:SoftRefLRUPolicyMSPerMB 每兆堆空间中软引用距上一次被引用后的存活时间，默认一秒。
-ea 启动断言，在断言失败时抛出异常信息
-Dsun.io.useCanonCaches 文件的全路径会默认缓存 30 秒，一般关闭这个功能避免无法及时拿到文件更新后的信息。参考。
-Djava.net.preferIPv4Stack 在获取 IP 地址时，优先获取 IPV4 地址，这在启用了 IPV6 地址时推荐使用，避免拿到 IPV6 地址，同样类似的有 -Djava.net.preferIPv6Addresses=true 配置
-XX:-OmitStackTraceInFastThrow 禁用 fast throw 优化，虽然该优化会让异常抛出变快，但该优化会清空异常的 message 和 stack trace 信息。

2、查看启动 GC 日志

准备工作做完了，就启动 idea 查看日志。启动参数里没有指定新生代的垃圾收集器，默认会使用 ParNew，老年代是 CMS，截取日志中发生了的一次 CMS GC，

统计了一下，一共发生了 15 次 CMS GC，一次 System.gc() 引起的 Full GC，通过命令 jstat -gcutil <pid> 后结果如下，

31 次 FGC 刚好是 15*2 + 1，和日志的内容展示符合。其中 322 次 YGC 用时 2.193s，平均 6.8ms，31 次 FGC 用时 1.511s，平均 48ms。

3、分析原因

只是启动情况，FGC 的次数貌似有点过多了，就主要分析一下这块。启动参数没有指定新生代和老年代的大小，使用 jmap -heap <pid> 查看堆分配情况，其中，Eden 区 200M，S0 和 S1 都是 25M，即默认 8:1，老年代 500M，一共 750M。这个分配是 idea 根据参数在启动时经过多次堆扩容后的结果，我将这个结果作为初始配置，同时，日志显示触发了一次 System.gc() 导致的 FGC，引用一句话：GC 远比我们想象的聪明，所以什么时候触发 GC 最好还是交给它自己处理，所以把这个也干掉，修改如下配置，

清空 GC 日志，重启 idea 后再查看一下情况，

可以看出，由于指定了 -Xms 参数，启动时对空间不需要扩大，由新生代初始空间过小触发的 YGC，和老年代初始空间过小导致的 FGC 都明显少了很多。再分析一下第一次发生 FGC 的日志，

可以看出，老年代在触发 FGC 后，在 CMS 的 Initial Mark 阶段，老年代使用了 35531K，老年代总大小是 51200K，按理 JDK8 里面 -XX:CMSInitiatingOccupancyFraction=<n> 参数在未指定情况，默认值约为 92（参考），还远不及进行 FGC 的标准，包括后面两次 CMS GC，也是这样的情况。

4、进一步分析

从上面日志可以看出，触发 FGC 时，堆空间是足够的，并不是引起 FGC 的原因，考虑 JVM 的内存模型，把栈区、程序计数器区排除开，那最有可能的就是元空间了。元空间最大值一般都 1G 以上，一般都不会超过这个值，但在上面的启动配置里，没有设置元空间初始大小，和前面分析一样，元空间大小初始值可能偏小，启动过程中扩容导致 FGC，也是查看启动后已使用元空间大小，将比这个值的略大的值作为初始大小，于是增加下面启动参数，

符合预期，FGC 一次都没有了。考虑自己电脑内存较充足，把最大内存扩大到 1024m，新生代 400m，重启后 YGC 只有 5 次，FGC 0 次。
